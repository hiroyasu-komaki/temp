# スコアリング方法論と判断基準

このドキュメントは、ベンダー切替投資ポートフォリオの優先度スコアと予算配分の
**判断根拠**を定義する。コードはこの仕様に従って実装される。数値を変えたい場合は
コードではなくこのドキュメントとシナリオ設定（`input/scenario.json`）を更新する。

---

## 1. 問題の性質

各契約は固有の満了時期を持ち、その手前でしか切替を意思決定できない。切替コスト・
想定リターン・実行確度は不確実で、情報ネットワークにより随時更新される。したがって
これは静的な最適化ではなく、**情報更新のたびに再評価する逐次的資源配分**である。

手元資金（リボルビング型予算）を、一定期間内に複数案件へどう配分するかを、
「いつでも説明できる」形で決めることが目的。

---

## 2. スコアの構造：Score = V × U × R

スコアは3因子の**積**とする。積を採る理由は、どれか一つでも決定的に欠けたら
その案件は張るべきでないから。和では一因子がゼロでも他が高ければ立ってしまう。

### 2.1 V — リスク調整期待価値

```
net_i = est_return_i - switch_cost_i
V_raw_i = p_i * net_i * (1 - lambda * (1 - p_i))
```

- `p_i` : 実行確度（0..1）
- `lambda` : リスク回避度。0 なら純粋な期待値最大化。上げるほど低確度案件を割り引く。
- リスクの代理として `p_i(1-p_i)` を用いる（確度0.5付近が最も曖昧＝高リスク）。

V は案件群の中で相対化する（最大値で割って 0..1 に正規化）。金額の桁が大きい案件が
緊急度・熟度を無視して常に上位に来るのを防ぐため。

#### 2.1.1 実行確度 p_i の内訳（2因子の積）

`p_i` は単一の数字ではなく、性質の異なる2つの確率の積として扱う。
情報更新時にどちらを叩くべきかが変わるため、分離する。

```
p_i = contestability_prob_i * execution_prob_i
```

- `contestability_prob` (p^contest) : **切替の土俵が成立する確率**。
  代替ベンダーが現れ、比較検討が可能になる確率。
  前段の三問コンテスタビリティテスト（Q1特定可能性・Q2市場厚み・Q3スイッチングコスト）
  がそのまま推定フレームになる。
- `execution_prob` (p^exec) : **土俵成立後、現場が切替を実行しきる確率**。
  一時延長に逃げず移行まで完遂する確率。

**後方互換**: CSVに `contestability_prob` と `execution_prob` の両方があればその積を使う。
なければ従来の `exec_probability` 単独をそのまま p_i として使う（既存データが壊れない）。

#### 2.1.2 情報更新時に何を叩くか

| 事象 | 何が起きているか | 更新する変数 |
|---|---|---|
| 切替先が現れない（今回のみ） | 土俵が今回立たなかった | contestability_prob↓、est_return↓、return_sigma↓ |
| 切替先が構造的に不在（Single-source確定） | 土俵が原理的に立たない | is_active=0（対象外・交渉トラックへ） |
| 一時延長（初回） | 判断が先送りされた | expiry_months を延長分だけ後ろへ、extension_count+1 |
| 繰り返し延長（閾値超え） | 現場の実行意欲が構造的に低い | expiry_months更新 + execution_prob へ自動ペナルティ |

「切替先が現れない」ケースで return_sigma を**下げる**のは、
「低リターンであることが確実になった」ため。見込みは下がるが確からしさは上がる。
結果、その案件はスコア上、静かに沈んで捨てる候補に落ちる（正しい挙動）。

「一時延長」を p の低下として記録してはならない。実行されにくいのではなく
判断機会が先送りされただけ。expiry_months を更新すれば d_i が再び伸び、
緊急度 U が下がって自動的に温存へ戻る。ただし繰り返し延長は別で、
現場任せでは切り替わらないシグナル＝VMOが実行主導すべき対象として扱う。

### 2.2 U — 緊急度

```
d_i = expiry_months_i - switch_lead_months_i   # 着手デッドラインまでの月
U_i = 0                     if d_i < 0          # 着手不能（失効）
U_i = 1 / (1 + d_i / h)     otherwise           # 双曲型
```

- `h` : リードタイム基準（月）。緊急度の効き方の基準。
- `d_i < 0`（満了までに切替期間が確保できない）案件は U=0 でスコアゼロ。取りこぼしを可視化する。

### 2.3 R — 情報熟度

```
g_i = min(max(d_i, 0) / h, 1)      # 締切が遠いほど不確実性を重く見る
R_i = 1 - return_sigma_i * g_i
```

- `return_sigma_i` : 想定リターンの相対不確実性（0..1）。
- 締切が迫っているときは不確実性ペナルティを緩める（待つ時間がないなら不完全でも判断する）。
- 締切が遠いときは不確実性を厳しく見て温存に回す。

## 3. 束ね（ベンダー単位バンドル）

同一ベンダーの複数契約は、束ねて一括交渉すると交渉力が上がる。一方で確度は下がり、
満了時期は最早に律速される。束ねは諸刃であり、意思決定変数として扱う。

```
bundle_return = beta * sum(member_returns)      # beta>=1 : 束ねプレミアム
bundle_cost   = sum(member_costs)
bundle_p      = (prod(member_p)) ** (1/sqrt(n)) # 緩めた積（オールオアナッシング性）
bundle_sigma  = max(member_sigmas)              # 最も不確実な要素に律速
bundle_expiry = min(member_expiries)            # 最も早い満了に律速
bundle_lead   = max(member_leads)
```

**採用条件**: 束ねの V_raw が構成契約の個別 V_raw 合計以上なら束ねを採用し、
構成契約を候補集合から除外して束ねで置換する。下回るなら個別のまま。
どちらの判断をしたかは必ず出力に残す（説明可能性）。

## 3.5 延長ペナルティ（繰り返し延長の検出）

現場が契約を一時延長した場合、初回は expiry_months を更新するのみ（判断の先送り）。
しかし繰り返し延長される案件は、現場の実行意欲が構造的に低いシグナルであり、
execution_prob を自動で割り引く。

```
if extension_count >= ext_threshold:
    over = extension_count - ext_threshold + 1
    penalty = min(ext_penalty * over, ext_penalty_cap)
    execution_prob_effective = execution_prob * (1 - penalty)
```

- `ext_threshold` : ペナルティ開始の延長回数（既定 2）。
- `ext_penalty` : 閾値超え1回あたりの割引率（既定 0.15）。
- `ext_penalty_cap` : 割引の上限（既定 0.6）。過度に沈めすぎない。

これらは scenario.json に置き、現場の延長頻度に応じて調整する。
繰り返し延長で execution_prob が下がった案件は、
「現場任せでは切り替わらない＝VMOが実行主導すべき対象」として浮上させる意味を持つ。
出力には延長回数とペナルティ適用の有無を残す（説明可能性）。

---

## 4. 予算配分（第2層）

スコア降順に並べ、手元資金 B の制約下で配分する。

- **本コミット**: 確度 `p >= 0.6` かつ「使用可能額 = B × (1 - safety_margin)」に収まる案件。
- **予約枠**: 確度 `p < 0.6` の案件。手元資金 B の範囲で資金を仮押さえ。不成立なら次へ回す（リボルビング）。
- **予算不足**: 優先度は高い（score >= 0.1）が資金が届かない案件。**明示する**。
  これが追加資金の引き出し根拠になる。
- 失効案件（d_i < 0）と極小スコア（score < 0.05）は配分対象外。

---

## 5. 判定ラベル（verdict）

| ラベル | 条件 | 意味 |
|---|---|---|
| 失効 | d_i < 0 | 満了までに切替不能 |
| 即着手 | score >= 0.25 | 3因子が揃う。最優先 |
| 人間判断 | U >= 0.5 かつ R < 0.7 | 締切近いが情報不完全。人が最終判断 |
| 温存 | Vn > 0.3 かつ U < 0.4 | 有望だが時間あり。情報を待つ |
| 捨てる候補 | Vn < 0.1 | 期待価値が低い |
| 検討 | 上記以外 | 中間 |

---

## 6. 入力規律（重要）

`return_sigma` は最も入力を怠られやすい。しかし R の生命線はこの値である。
**想定リターンを一つ入れたら、その確からしさ（sigma）も必ずセットで入れる**。
3段階（高0.6 / 中0.35 / 低0.15 程度）の粗い評価で構わない。空欄を許さない。
