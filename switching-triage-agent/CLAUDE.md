# CLAUDE.md — VMO ベンダー切替投資エージェント

このプロジェクトは、保守・ライセンス契約の切替投資について、リボルビング型予算を
どう配分すべきかを計算し、**いつでも説明できる形で**優先順位と配分根拠を示すエージェント。

## このエージェントがやること / やらないこと

**やる**: 契約データの読み込み・検証、優先度スコア（V×U×R）計算、束ね評価、
予算配分、説明レポート生成（データ生成〜計算まで）。ブラウザで動く簡易シミュレーション
ダッシュボード（`index.html`）の実装・保守も対象（下記「ダッシュボード」参照）。

**やらない**: 断定的な投資判断。ダッシュボードは「感覚的に分かりやすい少数のパラメータ」
のみ調整可能にする設計とし、`lambda_risk` や `lead_time_h_months` のような効果の説明が
難しいパラメータは固定表示に留める（UIで露出しない）。

## ディレクトリ

- `src/Main.py` — エントリポイント。`input → mid → output` のパイプライン。
- `src/modules/` — loader / scoring / allocation / reporter。
- `direction/` — 判断基準と出力仕様。**数式や閾値の根拠はここが正**。
- `input/` — 契約CSV（`contracts.csv`）とシナリオJSON（`scenario.json`）。
- `mid/` — 中間ファイル（`scored.json` 等）。
- `output/` — 成果物（`allocation.json`, `report.md`）。
- `index.html` / `assets/` — ブラウザ用シミュレーションダッシュボード（下記参照）。

## 実行

```
python src/Main.py
python src/Main.py --scenario input/scenario_conservative.json
```

## ダッシュボード（index.html）

`index.html` をブラウザで開くと、V×U×R計算を画面上で試せる。Python本体
（`src/modules/scoring.py`）のスコアリングロジックを `assets/js/scoring.js` に
移植したもので、数式・閾値はPython側と一致させる（乖離させたら原則3違反）。

- `assets/js/data.js` は手書き禁止。`src/modules/dashboard_sync.py` が
  `python src/Main.py` 実行のたびに `input/contracts.csv` / `input/scenario.json`
  から自動生成する（loader検証済みのデータをそのまま書き出すため、CSVを
  二重パースしない）。手動での再生成コマンドは不要。
- UIで動かせるパラメータは意図的に絞ってある: 評価基準日・手元資金・安全余裕・
  束ね評価のON/OFF（と束ねプレミアム）・延長への厳しさ（`ext_threshold`/
  `ext_penalty`/`ext_penalty_cap`をプリセット化した1本のスライダー）。
  評価基準日は通常3〜6ヶ月以上先を見て使う項目のため、シナリオ設定の最上部に置く。
- `lambda_risk` と `lead_time_h_months` は説明が難しいため固定表示のみ
  （値は `input/scenario.json` で調整）。
- 評価基準日は本体パイプラインでは表示ラベルのみだが、ダッシュボードでは
  シミュレーションの意味を持たせるため「基準日をずらすと各契約の残月数が
  再計算される」独自ロジックを持つ（`assets/js/scoring.js` の `shiftedExpiry`）。
  本体との差異として認識しておくこと。
- **予算配分は自動アルゴリズムではなく手動選択**。本体パイプライン
  （`src/modules/allocation.py`）が行う p≥0.6/使用可能額判定による自動配分は、
  ダッシュボードには移植していない。優先度ランキングの各行に「対象」
  チェックボックスがあり、ユーザーが選んだ案件の切替コスト・p×net合計を
  そのままKPI（本コミット額・期待ネット・ROI・手元資金との差額）に反映する
  （`assets/js/main.js` の `selectedIds`）。ROIは総回収ベース
  （p×想定リターン÷本コミット額、損益分岐点100%）で計算しており、
  V計算に使う`net`（リターン−コスト）ベースの期待値とは分母・分子の取り方が
  異なる点に注意。「本コミット額」はアルゴリズムの自動判定結果ではなく、
  ユーザーの選択結果である点が `output/report.md` の同名の値と意味が異なることに注意。
- **束ね案件は採否に関わらず常にランキングに表示**する（`assets/js/scoring.js`
  の `scoreAll`）。個別契約と束ね案件を両方候補として並べ、ユーザーがどちらを
  選ぶかチェックボックスで決める設計。束ね価値と個別合計の比較（採用/不採用）は
  行わず、スコアそのものをランキング上で比較させる（解釈は読み手に委ねる）。
  本体パイプライン（`scoring.py`）は引き続き採否に応じて構成契約を置換する
  自動ロジックのままで、ここがダッシュボードとの意図的な差異。

## 重要な原則

1. **説明可能性が最優先**。各判定が3因子（期待価値・緊急度・情報熟度）のどれに
   起因するかを常に示す。ブラックボックスの最適解は出さない。
2. **入力規律**。`return_sigma` の空欄を許さない。R の生命線であり、loader が停止する。
3. **数式を変えるときは `direction/scoring_methodology.md` とコードを両方更新**する。
   コードだけ変えると根拠ドキュメントと乖離する。
4. **パラメータ調整はコードでなく `input/scenario.json`** で行う
   （λ=リスク回避度, β=束ねプレミアム, h=リードタイム基準, 予算, 安全余裕）。
5. **束ねは諸刃**として扱う。総リターンは増えるが確度は下がる。採否は必ず出力に残す。

## スキル（.claude/commands/）

- `/analyze` — 分析パイプラインを実行してレポートを生成。
- `/compare` — パラメータを振って複数シナリオを比較。
