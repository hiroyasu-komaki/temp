# 第4章 switching-triage-agent — 判断を自動化し説明可能にする

## 4.1 エージェントの役割

switching-triage-agent は、第3章の枠組みを実装した意思決定支援エージェントである。
契約データを入力すると、優先度スコアと予算配分を計算し、**いつでも説明できる形の
レポート**を生成する。可視化ダッシュボードの前段、すなわちデータ生成・スコア計算・
予算配分・レポート生成までを担う。

名称の「triage（トリアージ）」は、限られた資源を緊急度と見込みで選別する動きを表す。
各案件を「即着手 / 温存 / 人間判断 / 捨てる候補 / 失効」に振り分けることが中核である。

## 4.2 処理の流れ

```
input（契約CSV + シナリオJSON）
  → 読み込み・検証（入力規律の強制）
  → スコアリング（V×U×R と束ね評価）      → mid（中間データ）
  → 予算配分（本コミット・予約枠・予算不足） → output（配分結果）
  → 説明レポート生成                        → output（report.md）
```

## 4.3 出力に必ず含まれるもの

1. **優先度ランキング**：各案件の V・U・R を分解表示し、判定ラベルを付す。
2. **束ね判定**：同一ベンダーの複数契約を束ねた場合と個別の、どちらが有利かとその根拠。
3. **予算配分**：本コミット・予約枠・予算不足の3区分。
4. **追加資金の引き出し根拠**：予算不足で着手できなかった上位案件の合計額と筆頭案件。
   これがそのまま、次の資金引き出しの定量的根拠になる。

## 4.4 検証で判明した3つの実務的含意

サンプルデータで動かした結果、机上では見えなかった論点が実証された。

第一に、**初期20百万円では最優先案件一件でほぼ使い切る**。「一件集中か分散か」に対し、
データは「初期20百万円では分散する余裕はなく、最も確実な一件に集中せざるを得ない」と
答えた。これはリボルビング提案（実績を作って次を引き出す）の論理と完全に整合する。

第二に、**「予算不足で着手不能」の案件が明示される**。優先度は高いが資金が届かない
案件が可視化され、これが追加資金引き出しの定量的根拠になる。提案の核心が数字として
立ち上がる。

第三に、**束ねは諸刃**。同一ベンダーの一括交渉は総リターンを増やすが、確度が下がり
（オールオアナッシング性）、予算配分では単独高確度案件に劣ることがある。束ねるべきかは
交渉力の自己評価（束ねプレミアム β）次第である。

## 4.5 束ね（ベンダー単位バンドル）の扱い

同一ベンダーが複数契約を持つ場合、個別に切り替えるより束ねて一括交渉する方が
交渉力が上がる。一方で確度は下がり、満了時期は最早に律速される。エージェントは
「個別に攻めた場合の合計」と「束ねた場合」を両方計算し、有利な方を採用する。
どちらを選んだかは必ず出力に残す。
