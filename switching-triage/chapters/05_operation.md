# 第5章 運用と情報更新 — 確度をどう捉え、いつ更新するか

## 5.1 実行確度は2つの確率の積である

実行確度 p を単一の数字として扱うと、性質の異なる2つの事象が潰れて見えなくなる。
p は本来、次の2つの積である。

```
p = contestability（土俵成立の確率） × execution（実行しきる確率）
```

- **contestability（p^contest）**：切替の土俵が成立する確率。代替ベンダーが現れ、
  比較検討が可能になる確率。三問コンテスタビリティテスト（Q1特定可能性・Q2市場厚み・
  Q3スイッチングコスト）がそのまま推定フレームになる。
- **execution（p^exec）**：土俵成立後、現場が切替を実行しきる確率。一時延長に逃げず
  移行まで完遂する確率。

情報ネットワークで捕捉するたびに、どちらを更新すべきかが変わる。だから分離する。

## 5.2 情報更新時に何を更新するか

| 事象 | 何が起きているか | 更新する変数 |
|---|---|---|
| 切替先が現れない（今回のみ） | 土俵が今回立たなかった | contestability↓、想定リターン↓、不確実性↓ |
| 切替先が構造的に不在（Single-source確定） | 土俵が原理的に立たない | 対象外へ（交渉トラックに移管） |
| 一時延長（初回） | 判断が先送りされた | 満了時期を延長分だけ後ろへ、延長回数+1 |
| 繰り返し延長（閾値超え） | 現場の実行意欲が構造的に低い | 満了時期更新 + execution へ自動ペナルティ |

## 5.3 ケース1：切替先が現れない

代替ベンダーを当たったが有効な移行先が存在しない/応じない場合、contestability を
下げる。ここで止めてはならず、切替先がないなら想定リターンの前提も崩れているため
リターンも下方修正する。

さらに重要なのは、**このケースは不確実性を下げる方向にも働く**点である。「切替先が
ない」という事実が判明したこと自体は情報の増加であり、「低リターンであることが
確実になった」ことを意味する。結果、その案件はスコア上、静かに沈んで捨てる候補に
落ちる。土俵が立たない案件に資金を張り続けないための、正しい挙動である。

「今回は現れなかった」のか「構造的に存在しない」のかは区別する。前者は次の満了
サイクルで再浮上しうるので案件を残す。後者は Single-source 確定であり、切替仮説を
非活性化し、「現行ベンダーとの交渉」という別トラックに移す。

## 5.4 ケース2：現場判断で一時延長

現場が契約を一時延長したとき、起きているのは「実行確度が下がった」のではなく、
**着手デッドラインが先送りされた**ことである。したがって更新すべきは確度ではなく
**満了時期そのもの**。これを確度の低下として記録すると、モデルは「この案件は
実行されにくい」と誤学習する。実態は、実行されにくいのではなく判断機会が先送り
されただけである。満了時期を更新すれば緊急度が下がり、案件は自動的に温存へ戻る。

ただし延長には警告シグナルの側面もある。**繰り返し延長される案件**は、現場の切替
意欲が構造的に低いことのシグナルであり、execution を下げる。一度目の延長は満了時期
更新のみ、二度目以降は execution も下げる二段構え。

この「繰り返し延長の検出」は提案の核心に触れる。現場が延長で逃げ続ける案件こそ、
VMOが「実行責任を引き受ける」と宣言した対象そのものである。延長回数が可視化
されれば、「この案件は現場任せでは永遠に切り替わらない。VMOが主導すべき」という
判断根拠になる。

## 5.5 入力規律

想定リターンの確からしさ（return_sigma）は最も入力を怠られやすいが、情報熟度 R の
生命線である。**想定リターンを一つ入れたら、その確からしさも必ずセットで入れる**。
エージェントはこの空欄を機械的に弾き、運用上も規律を保つ。3段階の粗い評価で構わない。

## 5.6 握るべき調整点

- **リスク回避度 λ**：手堅く行くか大きく取りに行くか。VMOとIT組織長で握る唯一の
  思想的パラメータ。「一件集中か分散か」という本来もめる論点を、λ という一つの
  合意事項に還元できる。
- **延長ペナルティの閾値**：現場の延長頻度に応じて調整する。延長が常態化している
  環境なら閾値を上げる必要がある。実データを何サイクルか回して体感と合わせる。
