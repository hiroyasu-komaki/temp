# 様式2 調達仕様書 — 本番実装に向けた設計アドバイス

対象: `Form2_Procurement_Specification.html`（現行モックアップ）
読み手: 本番Webアプリケーション開発チーム
前提スタック: モダンWeb汎用（React/Next.js + Node REST + PostgreSQL + OIDC）
作成: UI/UX・アーキテクチャアドバイザー

---

## 0. 結論（先に要点）

現行モックは「1枚のHTMLに全項目を並べ、契約類型で表示を出し分ける」構造です。デザイン自体は堅実ですが、トライアルで出た4つの不満（**項目が多い／どこに何を書くか分からない／テキスト入力ばかりで手間／途中保存できない**）は、いずれもこの「静的1枚もの・状態を持たない」構造から必然的に生じています。UIの微調整では解けません。

本番では次の3点を設計の土台に据えることを推奨します。

1. **プログレッシブ開示** — 全項目を一度に見せず、類型選択→必要なセクションだけを順に案内する。「多い／迷子」はここで大半が消えます。
2. **サーバ状態を持つ下書き（autosave＋再開）** — 入力は常時サーバに保存され、いつ中断しても続きから書ける。「途中保存できない」を根本解決します。
3. **入力を「書かせる」から「選ばせる・生成する」へ** — 自由textareaを、構造化入力・選択肢・AI下書きで補助する。「テキストばかりで手間」を軽減します。

以下、診断 → UI/UX方針 → 技術選定 → データモデル → AI支援 → ロードマップの順で具体化します。

## 1. 現行モックの診断（フィードバック → 構造原因）

| ユーザーの不満 | 現行モックの該当構造 | 根本原因 |
|---|---|---|
| 入力項目が多くて面倒 | セクション0〜14を単一ページに縦積み。類型選択で `section.hidden` を付け外しするが、表示された中は依然として全項目 | 「今このユーザーが埋めるべき最小集合」を提示していない。表示/非表示は粗く、1画面の情報密度が高い |
| どこに何を入力すればよいか分からない | `.guide` の説明文と `placeholder` の長い例文で補っている | ナビゲーション（現在地・全体像・進捗）が無い。0-3のセクションマップは静的リンクのみで「済/未/必須」の状態を持たない |
| テキスト項目ばかりでキーボード入力が手間 | 主要セクション（1,2,3,4,5,6,7）がほぼ `<textarea>` の自由記述。SLA・非機能・予算なども結局フリーテキスト | 半構造化できる項目（性能値・稼働率・RTO/RPO・金額・期日）まで自由記述に委ねている。定型はフォーム部品にできる |
| 途中で保存できないのは不親切 | `<script>` に `localStorage` も `fetch` も無い。リロードで全消失 | 永続化層が存在しない。純粋な表示モックのため状態管理・保存・再開の概念が無い |

補足として、現行モックには本番化にあたり以下も未実装です（設計時に織り込むこと）。

- 入力検証（必須・形式・整合性）と、VMOが最終判定する「見積可能性」の事前セルフチェック
- ファイル添付の実体（アップロード先・容量制限の実装・ウイルススキャン・形式検証）。現状は `<input type=file>` の見た目のみ
- 認証・権限（作成者／所属長承認／VMO）、監査ログ、案件番号の自動採番
- 多言語（JA/EN）はDOMの `data-ja/data-en` 差し替えで実装されており、本番のi18n基盤に載せ替えが必要

## 2. UI/UX 改善方針

### 2-1. プログレッシブ開示（「多い」「迷子」への回答）

**単一の長大フォームを、ステップ型（ウィザード）＋左ナビの構成に変える。**

- 最初に **0. 案件情報 → 0-2 契約類型** だけを提示する。ここは全体の分岐点なので、他を見せない。
- 類型を選ぶと、既存の `RULES`（`A:{secs:[…]}` などの表示ルール）をそのままサーバ側/フロント側の「ステップ定義」に流用し、**そのユーザーに必要なステップだけ**を左ナビに生成する。この分岐ロジックは既にモックに存在するので、資産として活かせます。
- 1画面＝1セクション（またはサブセクション）に絞る。スクロール量を「そのセクション分」に限定するだけで体感の重さが大きく下がります。
- 上部に **進捗インジケータ**（例: 7ステップ中3完了）、左に **ステップ一覧＋状態バッジ**（未着手／記入中／完了／要確認）を常時表示。0-3のセクションマップを「静的リンク」から「状態を持つナビ」へ格上げする形です。
- ただし調達文書は全体を俯瞰したいニーズもあるため、**「全体をプレビュー／印刷用の1枚表示」モードを併設**する（現行の "全セクションを表示" と `@media print` はそのまま活かす）。ウィザードは入力用、1枚表示はレビュー・PDF出力用、と役割を分ける。

### 2-2. 入力負担の軽減（「テキストばかり」への回答）

原則: **定型・半定型は部品化し、本質的に自由な記述だけをtextareaに残す。**

- **半構造化できる項目をフォーム部品へ**
  - 非機能要件（4-1）: 「性能・可用性・セキュリティ・規制・保守」を、まず**チェック該当 → 該当項目だけ数値/選択入力**（応答秒数、稼働率%、RTO/RPO時間、監査ログ保管年数など）に。自由記述は「補足」欄に降格。
  - SLA（4-2）: 指標・目標値・測定方法・未達措置を**行テーブル**（追加/削除可能な行）にする。現状の1つの大きなtextareaより、応札者にも読みやすく、後段のデータ活用が効く。
  - 予算（5-2）: 既に費目テーブルがある。**数値フォーマット（3桁区切り）・全角半角補正・電卓不要の自動合計**を付ける。
  - 期日/マイルストーン（5-1, 12）: `type=text` を **日付ピッカー**へ。
- **選択肢＋「その他」パターンの徹底**: 「不足情報の補い方」「契約形態」「提供条件（無条件/NDA後）」のように、既にチェック/ラジオ化されている良い部分を全セクションに広げる。自由記述は選択肢で拾えない例外だけに使わせる。
- **AIによる下書き生成（詳細は §5）**: 目的・背景（1）、現行環境の補足（2-1）、要求事項（3）など、本質的に自由記述な項目は、箇条書きメモや添付資料からAIが初稿を生成→人が直す方式にする。ゼロから文章を書く負担が最も大きい箇所なので、効果が高い。
- **入力を「その場の最小」に**: プレースホルダの長い例文は良いガイドだが、実際の入力欄に常時見えていると圧迫感がある。例文は「例を見る」トグルや脇のガイドパネルに寄せ、入力欄自体は軽く見せる。

### 2-3. 保存と再開（「途中保存できない」への回答）

- **常時オートセーブ**を標準にする。ユーザーに「保存ボタンを押す」責任を負わせない。入力停止から数秒（デバウンス）でサーバへ差分保存し、ヘッダに「◯分前に保存済み」を表示。
- **明示的な「下書き保存」ボタン**も併置（心理的安心・オフライン復帰時の手動トリガ用）。
- **案件（ドラフト）単位のリスト画面**を用意。「作成中」「VMO確認待ち」「差戻し」「承認済み」などのステータスで一覧・再開できる。
- ブラウザ障害対策として、送信失敗時のローカル退避（IndexedDB）→復帰時サーバ同期、を保険で持たせるとなお良い（必須ではない）。
- **楽観的ロック**（後述のバージョン列）で、同一ドラフトの多重編集や上書き事故を防ぐ。

### 2-4. 迷子をさらに防ぐ仕掛け

- 各セクション冒頭に「このセクションは何のためにあるか（応札者が何を判断するための情報か）」を1〜2行で明示。現行の `.guide` の思想は良いので、より短く要約して常設。
- **必須/任意の明確化**とインライン検証。空欄のまま次へ進めるが、未完は左ナビに黄色バッジで残す（強制ブロックはしない＝下書き文化を尊重）。
- **提出前のセルフチェック**: VMOが判定する7項目（目的・背景／現行環境／要求事項／非機能／制約／受入基準／前提と責任分界）について、「第三者が見積リスクを織り込める記述か」をAIが事前レビューし、弱い箇所を指摘（§5）。VMO差戻しの往復を減らす。

### 2-5. アクセシビリティ / 基本品質

- キーボード操作・フォーカス管理・`aria` 属性（現行はラベル紐付けが弱い）。スクリーンリーダ対応。
- レスポンシブは現行の `@media(max-width:820px)` の思想を踏襲しつつ、ウィザード化でモバイルはさらに扱いやすくなる。
- カラーコントラスト・フォーカスリング（現行の `outline` 指定は良好）を基準準拠で。

## 3. 技術選定（モダンWeb汎用）

### 3-1. 全体像

```
[Browser]
  Next.js (React) SPA/SSR
   ├ フォーム: React Hook Form + Zod（型・検証を共有）
   ├ サーバ状態/autosave: TanStack Query（+ 楽観更新）
   ├ i18n: next-intl or i18next（data-ja/data-en を辞書へ移行）
   └ AIチャット: ストリーミングUI（SSE/WebSocket）
        │  REST/JSON (OpenAPI 定義)
        ▼
[API]  Node.js — NestJS（推奨）または Fastify
   ├ 認証: OIDC/OAuth2（社内IdPとSSO）+ RBAC
   ├ ドラフト/提出/承認のワークフロー
   ├ ファイル: 署名付きURLで直接アップロード + スキャン
   └ AIオーケストレーション（LLM呼び出し・RAG・ガードレール）
        │
        ▼
[Data]
   ├ PostgreSQL（案件・回答・バージョン・監査）
   ├ Object Storage（S3互換：添付ファイル本体）
   └ （任意）ベクトルDB / pgvector（AI検索用）
```

### 3-2. フロントエンド

- **フレームワーク**: Next.js（React）。SSR/ルーティング/i18n/APIルートが揃い、社内業務システムに扱いやすい。SPA単体でも可だが、Nextを推奨。
- **フォーム基盤**: **React Hook Form + Zod**。Zodスキーマを1つ定義し、**フロントの検証とバックの検証を同一スキーマから共有**する（`zod` を API 側でも利用）。項目数が多いこのフォームでは、スキーマ駆動で「フィールド定義→レンダリング→検証→保存」を統一するのが保守性の鍵。
- **スキーマ駆動レンダリング**: セクション/項目の定義（ラベル・型・必須・表示条件・類型依存）をJSON定義に外出しし、UIはそれを解釈して描画する構成にすると、様式の改訂（項目追加・条項変更）を**コード改修なしに設定変更で**回せる。現行モックの `CATS` `RULES` `SECNAME` は、まさにこの定義の原型。
- **状態/通信**: TanStack Query でサーバ状態・autosave・再取得を管理。楽観更新でUX滑らかに。
- **i18n**: 現行の `data-ja/data-en`・`data-ph-*` を辞書ファイル（キー→JA/EN）へ機械的に移行。`html[lang]` 切替の思想は踏襲。

### 3-3. バックエンド

- **フレームワーク**: **NestJS**（構造化・DI・OpenAPI生成・大規模保守向き）を第一候補。軽量志向なら Fastify + Zod。
- **API様式**: REST/JSON、**OpenAPIで契約を先に定義**。ドラフトの部分更新は `PATCH`（JSON Merge Patch）でセクション/フィールド単位に。
- **ORM**: Prisma または Drizzle。マイグレーション管理必須。
- **ワークフロー**: ステータス機械（作成中→提出→VMO確認→承認/差戻し）をサーバ側で管理。状態遷移と権限をセットで実装。

### 3-4. 認証・認可・監査（調達システムとして重要）

- **認証**: OIDC/OAuth2 で社内IdPとSSO。汎用構成のため特定IdPに依存しない実装（Keycloak/Auth0/社内OIDC いずれも差し替え可能に）。
- **認可**: RBAC。ロール例＝**作成者 / 所属長（承認）/ VMO（発行前チェック）/ 閲覧**。セクション14の署名・承認はこのロールに対応させる。
- **監査ログ**: 誰が・いつ・どの項目を・どう変えたかを追記型で記録。調達の公平性・説明責任に直結。評価項目（11）の「VMO承認をもって凍結」「開札後変更不可」は、**サーバ側でフィールドロック＋変更履歴**として実装する（UIだけの制御にしない）。
- **採番**: 案件番号・受付番号はサーバ発番（現行モックの「システムが自動付与」記述と一致）。

### 3-5. ファイル添付

- **直接アップロード**: API経由でなく、**署名付きURLでObject Storageへ直接PUT**（大容量・添付多数に耐える）。
- **容量上限**: 現行は「別途定める」。サーバ側で**合計サイズ上限を強制**し、フロントの `.quota` バーはサーバの返す実測値で駆動。
- **安全性**: ウイルススキャン、MIME/拡張子検証、ダウンロード時の権限チェック、NDA条件付き資料（13の提供条件）のアクセス制御。

### 3-6. autosave の具体設計

- 入力イベント→**デバウンス（例1〜2秒）→ `PATCH /drafts/{id}`（変更フィールドのみ）**。
- レスポンスに `version` と `updatedAt` を含め、フロントは楽観ロック。競合時は409で「新しい版があります」を提示。
- サーバは各PATCHを**バージョン付きで履歴保存**（誤操作の復元・監査に有効）。

## 4. データモデル（PostgreSQL 案）

正規化と柔軟性のバランスとして、**「案件・提出のメタは列で正規化、回答本体はJSONBで柔軟に」** のハイブリッドを推奨。様式改訂に強く、かつ検索・集計したい列は正規化できる。

```
cases                 案件
  id (uuid, PK)
  case_no             サーバ発番
  title
  contract_period
  primary_type        例 "C-1"
  secondary_types     text[]        副類型
  status              draft|submitted|vmo_review|approved|returned
  created_by, dept
  created_at, updated_at

case_answers          回答本体（ドラフト実体・バージョン管理）
  id (PK)
  case_id (FK)
  version (int)                     楽観ロック/履歴
  data (jsonb)                      セクション1〜14の回答（スキーマ駆動）
  is_current (bool)
  updated_by, updated_at

attachments           添付
  id (PK), case_id (FK), section_key
  file_name, size_bytes, mime, storage_key
  scan_status, provision_condition  無条件/NDA後
  uploaded_by, uploaded_at

evaluations           評価項目・配点（11）
  id, case_id, criterion, weight, viewpoint, locked (bool)

clauses               標準条項の適用状態（8-1 / 8-1-2）
  id, case_id, clause_code, applied (bool), reason

audit_log             監査
  id, case_id, actor, action, field_path, before, after, at

users / roles / case_roles   認証・権限
```

補足:
- **回答をJSONB**にすることで、様式のセクション構成をフロント/APIの**スキーマ定義**（§3-2）と一致させられる。集計・検索が必要な項目（予算合計、類型、稼働率など）は生成列やビューで抜き出す。
- **凍結・ロック**（評価項目、標準条項）は `locked` / 状態と監査ログで担保。
- **バージョン**は autosave の履歴・差戻し時の比較に活用。

## 5. AI チャット / 入力支援の設計

自由記述の負担軽減とVMO往復削減に、AIが最も効く領域です。ユースケースを絞って導入します。

### 5-1. ユースケース（優先順）

1. **下書き生成**: ユーザーの箇条書きメモや添付資料（構成図・稟議書等）を入力に、目的・背景（1）／現行環境補足（2-1）／要求事項（3）／前提と責任分界（7）の**初稿を生成**。人は直すだけ。ゼロ起筆の負担が最大の箇所に当てる。
2. **見積可能性のセルフレビュー（提出前チェック）**: VMOが判定する7項目について、「第三者が見積リスクを織り込めるか」の観点でAIが弱点を指摘（例:「現行環境の規模＝利用者数・データ量が未記載。応札者は前提を置けません」）。VMOの差戻しをユーザー側で先回りして潰す。
3. **文脈ヘルプ**: 「この欄には何を書けばいい？」に、そのセクションのガイドと良い記入例で回答。現行 `.guide` の内容をAIの知識源にする。
4. **要件の言い換え支援**: 「手段でなく結果で書く」（3-1の原則）へのリライト。「◯◯画面を作る」→「◯◯ができる状態」への変換提案。
5. **類型選択の相談**: 「これはSaaSかシステム開発か」など、契約類型（0-2）の判断を対話で補助。

### 5-2. アーキテクチャ

- **RAG**: 記入ガイドライン・標準条項・過去の良質な仕様書・VMOチェック基準を知識ベース化（pgvector等）。AIの回答/生成をこの社内基準に接地させる。
- **オーケストレーション**: API側にAI層を置き、LLM呼び出し・RAG検索・出力整形・ガードレールを担う。フロントへは**ストリーミング（SSE）**で逐次表示。
- **書き戻し**: AIの生成結果は「提案」としてプレビュー→ユーザーが承認して初めて該当フィールドへ反映。勝手に確定しない。

### 5-3. ガードレール（調達文脈で必須）

- AIは**下書き・助言まで。確定・承認・提出はしない**（人の責任を残す）。
- 生成物には「AI下書き（要確認）」の由来を表示し、監査ログにも記録。
- **評価項目・配点や標準条項の凍結ルールをAIに破らせない**（凍結後の書き換え提案を出さない）。公平性に関わる領域はAIの権限外に。
- 添付資料・案件情報を外部LLMに渡す場合の**データ取扱い（社内基盤/契約・マスキング）**を情報セキュリティ部門と合意。GxP/個人情報/機密の観点は、扱う案件の性質上、最初に詰めること。

## 6. 実装ロードマップ（段階導入）

**フェーズ1 — 基盤（保存・再開・認証）**
モックのフィードバックのうち最重要の「保存できない」をまず解消。スキーマ駆動フォーム化＋autosave＋SSO＋ドラフト一覧。既存の類型分岐（`RULES`）をステップ定義に移植。

**フェーズ2 — 入力負担の軽減（構造化）**
非機能・SLA・予算・期日などを部品化。バリデーション、ファイルアップロード実体、提出→VMO承認ワークフロー、監査ログ、条項/評価の凍結。

**フェーズ3 — 迷子対策の仕上げとAI支援**
ウィザードナビの状態管理・進捗、提出前セルフチェック、AI下書き生成・文脈ヘルプ・RAG。ガードレールとデータ取扱いの合意を前提に段階リリース。
